# Atlas Demonstration Villa — Project Timeline

This is the narrative behind the Atlas Canonical Demo Project (ACDP),
generated by `backend/scripts/seed_demo_project.py`. It explains the
story the generator implements, so anyone reading the seeded data (or
maintaining the generator) understands why it looks the way it does —
this is not random filler data.

## The Project

**Atlas Demonstration Villa** — a luxury G+2 residence in New
Chandigarh, ~9,500 sqft built-up, executed by Studio Neoteric for a
private client, ₹3.6 Crore budget, 18-month build. As of "today" (the
day the seed script is run — see "How time works" below) the project is
approximately 95% complete: structure, MEP, and most finishes are done;
final snagging and client handover are the only work still in flight.

## The Site — Six Zones, One Project

A real villa plot isn't one uniform structure. ACDP models this as one
Atlas project containing six sites (zones), each with its own
construction pace and (mostly) its own subset of phases:

| Zone | What it is | Phases that apply |
|---|---|---|
| Main Residence | The G+2 primary structure | All 15 phases |
| Guest House | Smaller structure, north wing | All except Landscape, Handover |
| Basement & Utility Block | Sub-level, services-heavy | Earthwork through Testing, no Masonry/Finishes/Facade |
| Boundary Wall & Gate | Perimeter | Earthwork, Foundation, Masonry, Painting only |
| Landscape & Pool | Rear garden, pool | Earthwork, Waterproofing, MEP, Landscape only |
| Servant Quarters | South wing, smaller structure | Structure + finishes, no Facade/Landscape |

This is why the workflow isn't 6 identical copies of the same activity
list — a boundary wall never gets an RCC Structure phase, the basement
never gets a Facade phase, and only the Main Residence carries the
Client Handover phase (there's exactly one handover for the project as
a whole).

## The Phases

Fifteen phases, each mapped to construction reality and to the
Construction Reasoning Engine's own, unmodified stage-classification
vocabulary (engines/reasoning_projections.py's _STAGE_KEYWORDS) so
CRE's stage inference, health scoring, and look-ahead reasoning work
meaningfully on this data — the generator writes activity names the
classifier already understands, it does not change the classifier:

| ACDP Phase | CRE Stage | Keyword that matches |
|---|---|---|
| Earthwork | excavation | "excavat", "earthwork" |
| Foundation | foundation | "pcc", "footing", "foundation" |
| RCC Structure (incl. Roof) | rcc_structure | "rcc", "column", "beam", "slab" |
| Masonry | masonry | "masonry", "brick", "block work" |
| Waterproofing | waterproofing | "waterproof" |
| MEP | mep | "electric", "plumbing", "hvac", "conduit" |
| Flooring, False Ceiling, Painting, Joinery, Facade | finishes | "plaster", "paint", "tile", "door", "window", "cladding" |
| Testing | testing_commissioning | "testing" |
| Snagging | testing_commissioning | "snag" |
| Client Handover | handover | "handover" |

Landscape is the one phase with partial classification. Four of its
nine activities (Swimming Pool Shell Construction, Garden Soft-scaping
& Turfing, Garden Hardscaping & Pathways, Boundary Planting & Hedging)
have no real trade-keyword overlap with any CRE stage — the classifier
has no "landscape"/"garden"/"turf" keyword, a real gap in the current
classifier, not something this seed script should paper over. The
other five (pool filtration plumbing, pool tiling, the outdoor water
feature, outdoor lighting, outdoor furniture) genuinely are other
trades (plumbing, flooring, electrical) and correctly classify into
mep/finishes exactly as any real activity with those names would. If a
future sprint adds landscape awareness to the classifier, the four
currently-unclassified activities will pick it up automatically with
zero changes here.

## How Time Works

The generator computes everything in story days (0 = project start,
548 ≈ project start + 18 months) and then maps every day to an absolute
timestamp anchored so day 520 lands on "now" — whenever the seed
script is actually run. This means:

- The dataset always looks like an 18-month-old, still-active project,
  regardless of when you seed it.
- Re-seeding next month produces a story that still ends "today" —
  the narrative is evergreen, not frozen to a fixed calendar date.
- The handful of days between 520 and 548 represent the small remaining
  slice of work (final snagging, handover) that hasn't happened yet
  in the story — those activities are not_started/in_progress, not
  completed, which is what makes "95% complete" literally true rather
  than just a label.

## Phase Calendar (Overlap Is Deliberate)

Real construction phases overlap — MEP first-fix starts in one zone
while masonry is still finishing in another; painting starts on the
ground floor while joinery is still being fitted upstairs. The
generator uses a hand-tuned calendar of (start_day, end_day) windows
per phase (see PHASE_WINDOWS in seed_demo_project.py) rather than a
computed critical-path schedule — deliberately: a full scheduler is a
different, much larger feature, and a hand-authored, realistic-looking
calendar is sufficient (and far more auditable) for a demonstration
dataset's purpose. Each zone gets its own slice of a phase's window (so
multiple zones visibly progress in parallel), with a small amount of
seeded random jitter so it doesn't look mechanically uniform.

## The Story Beats

Within that calendar, the generator writes the specific texture the
brief asked for:

- Weather delays, material shortages, labour shortages, vendor delays,
  drawing revisions — modelled as a ~8% chance per activity in the
  phases where real delays cluster (RCC Structure: steel/weather; MEP:
  vendor lead times; Flooring & Facade: drawing revisions and client
  decisions). A delayed activity gets a real "blocked" workflow status
  transition, a linked site_issue operational item explaining the
  specific reason (drawn from a realistic reason bank — "unseasonal
  rain," "cement delivery delay," "steel price renegotiation," etc.),
  and then genuinely resumes (in_progress again) — a real, resolved
  blocker in the item's history, not just a label.
- Recovered schedule — every delayed activity's extended duration is
  absorbed within its phase's calendar window (the window is sized
  with slack for exactly this), so the project's overall 95%-at-day-520
  outcome holds even though individual activities ran over.
- Quality inspections & successful milestones — every activity flagged
  requires_inspection in the catalog (pre-pour checks, waterproofing,
  electrical/plumbing/fire-fighting testing, snagging rounds) carries
  that flag through to its generated workflow activity, exactly like
  any real Atlas project.
- Safety observations — generated on ~22% of activities across all
  zones (scaffolding, missing PPE, unbarricaded excavation edges,
  exposed panels), each resolved the same day it's raised, matching how
  a well-run site actually operates.
- Client decisions & approval bottlenecks — Client Approval Workflow
  requests (see below) are concentrated in Painting, Facade, Flooring,
  Joinery, False Ceiling, and Landscape — exactly where a real client is
  asked to choose finishes, colours, and fixtures — at a rate that
  produces a genuine backlog some months and a quiet month other times,
  not a perfectly even drip.

## Client Approvals

Every Client Approval Request is created through the real Client
Approval Workflow endpoint (request-approval), linked to a real
captured event — never a standalone, disconnected record. Client
decisions are distributed realistically, not uniformly:

- ~55% approved outright
- ~20% sent back with "Request Clarification" (a real, non-terminal
  question — see operations_engine.request_clarification)
- ~15% rejected, asking for revision
- ~10% left genuinely pending — a real, common "client hasn't decided
  yet" state, exactly what a live project looks like at any given
  moment

## AI Proposals

At roughly one checkpoint per week of the project (~55 events total,
spread across the whole timeline, not clustered at the start), the
generator runs the real intelligence_engine.generate_proposals_for_event
pipeline against a hand-authored, phase-appropriate structured analysis
(materials, labour, matching the activity and trade actually active at
that point) — the same pattern scripts/db_seed.py's own demo content
already established, just parameterised instead of a single fixed
example. Every proposal then receives one of the four real outcomes,
cycling deterministically so all four genuinely exist in the dataset:
accepted, rejected, modified (accepted with an edited priority/
description), and ignored (left pending — a real AI suggestion nobody
has acted on yet, exactly like production).

## CRE Reasoning Across History

reasoning_engine.run_reasoning() — completely unmodified — is called at
(approximately) each monthly checkpoint as the simulated timeline
advances, not just once at the end. Each call snapshots whatever state
genuinely exists in the database at that point in the simulated
history (however many activities have completed, however many
approvals are pending, etc. by that story-day) — this is what makes the
resulting reasoning_insights and construction_memory collections a
real evolution, not a single end-state analysis. Roughly 16-19 runs
result, each a real, independently-queryable reasoning_runs document.

One honest limitation: the reasoning run documents' own created_at
timestamps reflect when the seed script was executed (all clustered
together, since the script runs in a few minutes), not spread across 18
months — only the underlying evidence they reason over (events,
activities, operational items) carries genuinely 18-month-spread
timestamps. This is the correct, honest behaviour for "what would CRE
have said if asked at project-month-3" (the content of each run
reflects month-3 state) without needing to alter
reasoning_engine.run_reasoning's own timestamping — which this
project's constraints explicitly forbid touching.

## Reproducing This Document's Numbers

Running `python -m scripts.seed_demo_project` against a fresh database
produces (deterministically, every time — see the README):

- 361 workflow activities across 15 phases and 6 zones
- ~780 events (≈90 voice, ≈245 photo, ≈445 text)
- ~135 operational items, of which ~38 are client approval requests
- 56 AI proposals across all four decision outcomes
- 16-19 CRE reasoning runs spanning the full 18-month history
- A real, evolving Construction Memory record for every activity CRE
  observed complete during one of those runs
